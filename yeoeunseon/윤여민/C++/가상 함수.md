# 가상 함수(Virtual Function)

> [!summary]
> `Character*`가 실제로 `Warrior` 객체를 가리키고 있다고 생각해 보자.
>
> 이 포인터로 `Move()`를 호출했을 때 `Warrior::Move()`가 실행되게 만드는 기능이 **가상 함수(Virtual Function)**다.
>
> **한 줄 핵심: 기반 클래스 포인터나 참조로 호출해도 실제 객체에 맞는 함수를 실행한다.**

---

## 1. 왜 필요한가

`Warrior`가 `Character`를 상속하면 `Character*` 포인터나 `Character&` 참조로 `Warrior` 객체를 다룰 수 있다.

```cpp
Warrior Object;
Character* Ptr = &Object;
```

이때 코드에 적힌 포인터 타입은 `Character*`지만, 실제로 만들어진 객체는 `Warrior`다.

- **정적 타입(Static Type)**: 코드에 선언된 타입인 `Character*`
- **동적 타입(Dynamic Type)**: 포인터가 실제로 가리키는 객체 타입인 `Warrior`

일반 함수는 정적 타입을 보고 호출할 함수를 정한다. 가상 함수는 동적 타입을 확인해서 호출할 함수를 정한다.

```cpp
#include <iostream>

class Character
{
public:
    // 일반 함수
    void Attack() { std::cout << "Character Attack\n"; }
    
    // virtual이 붙은 가상 함수
    virtual void Move() { std::cout << "Character Move\n"; }
};

class Warrior : public Character
{
public:
    void Attack() { std::cout << "Warrior Attack\n"; }
    void Move() override { std::cout << "Warrior Move\n"; }
};

int main()
{
    Warrior Object;
    Character* Ptr = &Object;
    
    Ptr->Attack(); // 일반 함수: Character Attack
    Ptr->Move();   // 가상 함수: Warrior Move
}
```

`Ptr`의 타입은 두 호출 모두 `Character*`로 같다. 그러나 `Move()`에는 `virtual`이 붙어 있어서 실제 객체인 `Warrior`의 함수가 실행된다.

### 정적 바인딩 vs 동적 바인딩

**바인딩(Binding)**은 함수 호출문이 “어느 클래스의 함수를 실행할지” 결정되는 것을 뜻한다.

| 구분 | 일반 함수 | 가상 함수 |
| :--- | :--- | :--- |
| **바인딩 방식** | 정적 바인딩 (Static Binding) | 동적 바인딩 (Dynamic Binding) |
| **쉽게 말하면** | 실행 전에 결정 | 실행 중 실제 객체를 보고 결정 |
| **판단 기준** | 정적 타입 | 동적 타입 |

- **정적 바인딩(Static Binding)**: 컴파일할 때 호출할 함수가 결정된다.
- **동적 바인딩(Dynamic Binding)**: 프로그램이 실행될 때 실제 객체에 맞는 함수가 결정된다.
- **런타임 다형성(Runtime Polymorphism)**: 같은 부모 타입의 포인터나 참조로 호출해도 실제 객체에 따라 다른 동작을 하는 성질이다.

> [!note]
> 여기서 **런타임(Runtime)**은 프로그램이 실행 중인 시점을 뜻하고, **다형성(Polymorphism)**은 같은 호출이 객체에 따라 여러 형태로 동작한다는 뜻이다.

### 값으로 복사하면 다형성이 유지되지 않는다

가상 함수의 동적 디스패치는 기반 클래스 **포인터나 참조**로 파생 객체를 다룰 때 의미가 있다. 파생 객체를 기반 클래스 값으로 복사하면 파생 클래스 부분이 잘리는 **객체 슬라이싱(Object Slicing)**이 발생한다.

```cpp
Warrior Object;

Character& Reference = Object;
Reference.Move();        // Warrior::Move()

Character Sliced = Object;
Sliced.Move();           // Character::Move()
```

`Sliced`는 더 이상 `Warrior` 객체가 아니라, `Object`에서 `Character` 부분만 복사해 만든 별도의 `Character` 객체다.

## 2. vptr과 vtable의 동작

C++ 표준은 가상 함수의 구현 방식을 강제하지 않지만, 대부분의 컴파일러는 **vtable(가상 함수 테이블)**과 **vptr(가상 테이블 포인터)**을 사용해 동적 바인딩을 구현한다.

```mermaid
flowchart LR
    A["Character*가 Move 호출"] --> B["객체의 vptr 확인"]
    B --> C["Warrior vtable 접근"]
    C --> D["Warrior::Move 실행"]
```

- **vtable (Virtual Function Table)**: 가상 함수의 최종 재정의 함수 주소 등을 담는, 컴파일러가 만드는 표다. 일반적인 구현에서는 같은 동적 타입의 객체들이 공유한다.
- **vptr (Virtual Table Pointer)**: 객체 내부에서 해당 객체에 맞는 vtable을 가리키는 숨은 포인터다.

> [!note]
> `vptr`과 `vtable`은 널리 쓰이는 구현 방식이지만 C++ 표준이 요구하는 구조는 아니다. 다중 상속이나 가상 상속에서는 객체에 여러 `vptr`이 생기는 등 구조가 더 복잡해질 수 있다.

## 3. 객체 크기와 성능 비용

### 객체 메모리 변화와 패딩
일반적인 컴파일러 구현에서는 클래스가 다형적이면 객체에 vtable을 가리키는 숨은 포인터가 포함된다.

```cpp
class Normal { int Value; };
class Polymorphic { int Value; virtual void Func(); };

// 흔한 64비트 구현의 예:
// sizeof(Normal)      == 4
// sizeof(Polymorphic) == 16
```

- 포인터가 8바이트인 흔한 64비트 환경에서는 숨은 포인터와 정렬 패딩 때문에 위 예제의 `Polymorphic`이 16바이트가 될 수 있다.
- 가상 함수가 늘어나면 일반적으로 vtable 항목이 늘어난다. 단순한 단일 상속에서는 객체 크기가 그대로인 경우가 많다.
- 정확한 객체 크기와 `vptr` 개수는 ABI, 컴파일러, 상속 구조에 따라 달라지므로 `sizeof`로 확인해야 한다.

### 런타임 성능 비용
가상 함수는 일반 함수에 비해 약간의 런타임 오버헤드가 존재한다.

1. vtable에서 함수 주소를 찾은 뒤 호출하는 **간접 호출 비용**
2. 실제 호출 대상을 컴파일 시점에 알기 어려워 생기는 **인라이닝 제한**

> [!tip]
> 성능이 극도로 중요한 타이트한 루프 내부가 아니라면 대부분의 경우 가상 함수 호출 비용은 미미하다. 또한 컴파일러가 코드 분석을 통해 동적 바인딩을 일반 함수 호출로 바꾸는 **탈가상화(Devirtualization) 최적화**를 적용하기도 한다.

## 4. override와 가상 소멸자

### override 지정자

> **핵심: `override`는 부모의 가상 함수를 제대로 재정의했는지 컴파일러가 검사하게 한다.**

`override`가 없어도 함수 형태가 정확하면 재정의된다. 하지만 오타가 있거나 매개변수 타입, `const` 등이 다르면 재정의가 아닌 별개의 함수가 될 수 있다. `override`를 붙이면 이런 실수를 컴파일 오류로 알려준다.

```cpp
class Parent { virtual void Render(float DeltaTime); };
class Child : public Parent { void Render(double DeltaTime) override; }; // ❌ 컴파일 에러 발생
```

### 가상 소멸자 (Virtual Destructor)

> [!important]
> **가상 소멸자는 다른 객체의 소멸을 막는 기능이 아니다.**
>
> `Warrior` 객체 하나를 삭제할 때, 그 객체 전체를 `Warrior → Character` 순서로 빠짐없이 정리하게 한다.

```text
잘못된 이해
Character 객체 + Warrior 객체       ← 객체 두 개가 따로 존재하는 것이 아님

실제 구조
Warrior 객체 하나
└─ 그 안에 Character 기반 클래스 부분이 포함됨
```

상속으로 만들어진 `Character` 부분은 별도로 생성된 부모 객체가 아니다. `Warrior` 객체 하나를 구성하는 **기반 클래스 부분 객체(Base Class Subobject)**다.

객체 하나를 완전히 삭제하려면 바깥쪽인 `Warrior`부터 정리하고, 그 안에 포함된 `Character` 부분을 이어서 정리한다.

```text
Warrior 객체 하나 삭제
        ↓
Warrior::~Warrior()       자식에서 추가한 부분 정리
        ↓
Character::~Character()   상속받은 기반 클래스 부분 정리
```

이 과정은 같은 객체 하나의 내부를 순서대로 정리하는 것이다. 다른 `Character` 객체나 다른 `Warrior` 객체에는 아무 영향도 주지 않는다.

문제는 실제로 만든 객체가 `Warrior`여도 포인터에는 `Character*`라고 적혀 있을 수 있다는 점이다.

```cpp
class Character
{
public:
    virtual ~Character() = default;
};

class Warrior : public Character
{
public:
    ~Warrior() override = default;
};

Character* Ptr = new Warrior();
delete Ptr;
```

`Character` 소멸자가 `virtual`이므로 `delete Ptr`는 실제 객체가 `Warrior`라는 사실을 반영한다. 그 결과 같은 객체 안에서 `Warrior` 소멸자와 `Character` 소멸자가 올바른 순서로 실행된다.

> **규칙: 기반 클래스 포인터로 파생 객체를 삭제할 수 있다면 기반 클래스의 소멸자를 `virtual`로 선언한다.**

### 정의되지 않은 동작이란?

반대로 `Character` 소멸자가 가상이 아닌 상태에서 `Character*`로 `Warrior`를 삭제하면 **정의되지 않은 동작(Undefined Behavior, UB)**이다. 이는 C++ 표준이 실행 결과를 보장하지 않는다는 뜻이다.

흔히 파생 클래스의 소멸자가 실행되지 않아 파일, 메모리 같은 자원이 정리되지 않는 모습으로 나타날 수 있다. 하지만 “부모 소멸자만 실행된다”는 결과조차 표준이 보장하지 않으므로 그런 동작에 의존해서는 안 된다.

### 순수 가상 함수와 추상 클래스

구체적인 파생 클래스가 구현해야 하는 인터페이스는 순수 가상 함수로 표현할 수 있다.

```cpp
class Character
{
public:
    virtual ~Character() = default;
    virtual void Move() = 0;
};
```

`= 0`이 붙은 함수가 순수 가상 함수다. 순수 가상 함수가 하나라도 남아 있는 클래스는 **추상 클래스(Abstract Class)**이며 직접 객체를 만들 수 없다. 파생 클래스도 이를 재정의하지 않으면 계속 추상 클래스다.

## 5. 생성자와 소멸자에서의 가상 함수 호출 주의

> [!caution]
> 생성자와 소멸자에서 가상 함수를 호출하는 것은 문법적으로 가능하지만, 파생 클래스까지 동적 디스패치되지 않는다.
>
> - 기반 클래스 생성자가 실행될 때 파생 클래스 부분은 아직 초기화되지 않았다.
> - 따라서 기반 클래스 생성자에서 호출하면 현재 생성 중인 기반 클래스의 최종 재정의 함수가 실행된다.
> - 소멸 중에도 이미 소멸된 파생 클래스 쪽으로 호출되지 않고 현재 소멸 중인 클래스 기준으로 실행된다.
>
> 파생 클래스의 재정의 함수가 실행되기를 기대하는 코드는 작성하지 않는 것이 좋다. 특히 생성자나 소멸자에서 가상 디스패치가 순수 가상 함수에 도달하면 정의되지 않은 동작이므로 호출하지 않는다.

## 정리

- `virtual`: 실제 객체 타입을 기준으로 함수 호출
- 적용 조건: 기반 클래스 포인터나 참조로 파생 객체를 다룰 때 동적 디스패치
- 객체 슬라이싱: 파생 객체를 기반 클래스 값으로 복사하면 파생 클래스 부분과 다형성이 사라짐
- `vptr`: 객체가 자신의 vtable을 가리키는 포인터
- `vtable`: 가상 함수 주소를 저장하는 클래스별 테이블
- `override`: 가상 함수 재정의 실수를 컴파일 단계에서 검사
- 순수 가상 함수: 파생 클래스가 구현할 인터페이스 정의
- 가상 소멸자: 객체 하나를 실제 파생 타입부터 기반 클래스 부분까지 안전하게 삭제
- 비용: 일반적인 구현에서 객체 크기 증가, 간접 호출, 인라이닝 제한
